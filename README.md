# planning-mode

一個輕量級的 Claude Skill，只有一份 `SKILL.md`，不依賴任何額外的 script 或外部工具。

## 解決什麼問題

預設情況下，Claude 在對話裡聽到「我想做一個 XX」之後，常常會直接開始寫 code、建 artifact、甚至跳出完整的 architecture——使用者根本還沒講清楚需求細節，Claude 就已經自己猜完所有答案動手做了。

問題是，如果你的開發流程是「在 Claude.ai / app 規劃，在 Claude Code 實作」，這種無腦直接生產出來的程式碼或文件其實沒有用：

- 寫出來的 code 終究要搬到 Claude Code 重新生成一次，前面寫的等於浪費 token
- Claude 自己猜的命名、檔案結構、框架選擇，常常猜錯，要重講一次
- 規劃過程沒有留下任何「可以帶走」的紀錄，每次重開對話都要從頭講

這個 skill 就是用來把「規劃」跟「實作」這兩個階段切開，讓 Claude 在規劃階段乖乖停在規劃,該問的先問，最後產出一份結構清楚的文件，而不是自己生出一堆程式碼。

## 設計思路

整個 skill 用四層優先序規則組成，衝突時高層級蓋過低層級：

**Level 0 退出（覆蓋一切）> Level 1 核心準則 > Level 2 回應分流 > Level 3 產出規格**

1. **Level 0 — 退出** — 使用者說「結束規劃」等明確語句時立刻停用所有規則；話題自然離開專案範疇時也正常回應。退出判斷優先於其他任何一層。

2. **Level 1 — 核心準則** — 模式內永遠適用的三條底線：先問清楚再動手、把問題在對話裡問到有答案（不留「待確認事項」）、不寫完整實作 code。

3. **Level 2 — 回應分流** — 每則訊息先判斷「這個回答的終點是不是程式碼？」：
   - 不是（查詢、比較、原理）→ 直接回答，不需要包裝
   - 是（實作性質）→ 產出一段可以直接貼到 Claude Code 執行的 prompt

4. **Level 3 — 產出規格** — 規劃到一定階段，整理成 `plan_v{版號}.md`（新專案）或 `feature_{描述}.md`（既有專案加功能）。版號區分架構性變動（整數）跟細節調整（小數點），讓文件本身就帶有變更歷史的訊號。

## 跟單純丟 prompt 的差別

沒有這個 skill 時，使用者必須每次自己提醒 Claude「先別寫 code」「等等，先問我細節」。有這個 skill 之後，這個提醒變成 Claude 自己的預設行為——而且行為邊界寫得足夠具體（什麼算規劃內、什麼算規劃外、什麼時候升版號、衝突時哪一層優先），所以不會每次都要重新解釋一次規則。

## 使用方式

把 `planning-mode/SKILL.md` 放進你的 skills 目錄即可，不需要額外安裝任何 script 或依賴套件。

```
你的repo/
└── planning-mode/
    └── SKILL.md
```

## 適用場景

- 個人專案從零開始規劃，但實作要交給 Claude Code 執行
- 既有專案要加新功能，想要一份獨立、乾淨的功能規劃文件
- 不想要 Claude 在還沒問清楚細節前就自己假設答案動手做

## 不適用場景

- 單純想要 Claude 馬上幫你寫一小段 code（這個 skill 會先攔下來問問題，可能不是你要的速度）
- 跟「規劃一個專案」無關的一般技術問答（skill 設計上會自動判斷離開範疇，但如果你發現誤判，直接說「結束規劃」即可切回正常模式）
